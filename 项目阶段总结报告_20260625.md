## 4. 当前存在的问题

### 4.1 实物协议链路未闭环

这是当前第一优先级问题。ILA 能看到 A 端 TX 活动和 B 端 RX 脉冲，但 B 端无法稳定解出有效帧，导致 ACK 不返回，PS 侧最终表现为 `tx_retry_exhausted` 或早期的 `tx_dma_busy_timeout`。

可能方向包括：

- TFDU 输出极性与 RTL RX 采样预期不一致。
- 4PPM chip 宽度、采样窗口或 RX 判决门限不适合当前硬件波形。
- lane0/lane1 物理接线、A/B 方向、TX/RX 交叉关系与 RTL 映射不一致。
- TFDU SD/MODE/TX 使能时序或半双工收发方向切换不满足器件实际要求。
- 当前 ILA 采样显示的是电平活动，但协议前导码、符号边界或帧头无法被 RX FSM 正确锁定。

### 4.2 速率目标还没有实测闭环支撑

32/16 Mbit/s 目前有 raw PHY 模型证据，2lane 有仿真 payload 吞吐证据，但还没有：

- TFDU 实物有效 payload 吞吐。
- PS 到 PL 到红外再回 PS 的稳定吞吐。
- PC 到 PS 到 PL 到红外再回 PC 的端到端吞吐。
- 长时间连续传输下的吞吐和错误率统计。

### 4.3 8lane 仍是扩展目标，当前硬件验证集中在 2lane

RTL 和仿真已经有多 lane 扩展路径，但当前上板调试主要是 2lane。最终 8lane 接入、8lane half-duplex、4+4 full-duplex、lane 冗余/调度策略、资源/时序余量都还需要进一步验证。

### 4.4 PC-PS 网络闭环仍停留在离线和源码层

PC host client、PS TCP bridge 和 mock 协议测试已经具备基础，但尚未看到真实板级 TCP/DHCP 连接、断线重连、PC 下发数据、PS 上传红外数据的硬件证据。最终目标要求的 DHCP、静态 IP fallback、TCP 重连和上位机可视化/控制仍需实测。

### 4.5 构建与证据链需要进一步收敛

当前 reports 目录里既有早期失败日志，也有最新成功日志。需要在阶段验收资料中明确“最新有效版本”的 bitstream、XSA、ELF、BOOT.BIN、ILA CSV、UART log，避免旧失败日志和新成功日志混淆。

### 4.6 Vivado warning 和 DRC warning 仍需处理

最新实现没有 timing 和 route 阻塞问题，但仍存在：

## 5. 改进方向与下一阶段计划

### 5.1 先把单 lane 实物解码打通

建议下一阶段优先降低复杂度，先固定一个 lane，完成 A->B 单向有效帧解码和 ACK 返回：

1. 固定 lane0，记录 A_TX、B_RX、B_RX FSM debug、B_TX ACK、A_RX ACK 全链路 ILA。
2. 降低 PHY 速率，增大 `CNT_CHIP_MAX`，先验证低速能否解帧。
3. 增加或导出 RX FSM 关键信号：preamble detect、chip counter、symbol decode、frame type、crc_ok、rx_error_code。
4. 对 TFDU 输出极性做 A/B 试验，必要时在 RX 输入处参数化 invert。
5. 对 SD/MODE/TX 方向切换加保护时间，验证半双工器件切换是否影响 ACK。

通过标准建议先定为：`rx_ok > 0`、`tx_fail < sent`、至少一个完整 packet 能收到 ACK 并返回。

### 5.2 再恢复 2lane 并行和方向切换

单 lane 通过后再逐步恢复：

- lane0 独立通过。
- lane1 独立通过。
- lane0+lane1 同向 2lane 通过。
- A->B 与 B->A 方向分别通过。
- 2lane 非对称大/小流量模式通过。

每一步都保留 UART summary、ILA summary、CSV、bit/LTX 版本。

## 6. 阶段性评价

项目当前基础扎实，尤其是协议仿真覆盖、自动化脚本、Vivado 工程产物、PS/host 软件雏形和 ILA bring-up 流程都已经搭起来了。现在不再是从零验证可行性，而是进入典型的硬件 bring-up 阶段：真实器件波形、时序、极性、lane 映射和协议采样窗口需要逐项收敛。
